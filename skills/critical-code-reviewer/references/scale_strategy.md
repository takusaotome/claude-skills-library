# Scale Strategy (Code Review)

大規模コードベースのレビュー時に、効率的にホットスポットを特定し
重点レビューを行うための戦略。

---

## 発動条件

以下のいずれかに該当する場合にこの戦略を適用:
- 対象コード **500行超**
- 対象ファイル **5ファイル超**

## Phase A: ホットスポット抽出

Read/Grep ツールのみで実行可能な指標を使用してホットスポットを特定する。

### 一次指標（ツール不要、常に利用可能）

| # | 指標 | 閾値 | 検出方法 |
|---|------|------|---------|
| 1 | ファイルサイズ | 300行超 | Read で行数確認 |
| 2 | 関数/メソッド長 | 50行超 | `grep -n "def \|function \|class "` で関数位置特定 → 行数計測 |
| 3 | ネスト深度 | インデント4段超 | Grep でインデントレベル検出 |
| 4 | import 密度 | import 文10行超 | `grep -c "import\|from.*import\|require("` |
| 5 | 例外処理密度 | try/catch 5箇所超 | `grep -c "try:\|try {\|except\|catch"` |

### 二次指標（git 利用可能な場合のみ）

| # | 指標 | 検出方法 |
|---|------|---------|
| 6 | 変更頻度 | `git log --oneline -- <file> \| wc -l` 上位 |
| 7 | 最終変更日 | `git log -1 --format="%ci" -- <file>` 直近変更優先 |

### git が利用できない場合

一次指標のみでホットスポットを決定する。

## Phase B: 2段階レビュー

### Stage 1: ホットスポット重点レビュー

ホットスポット（上位5-10ファイル/関数）に対して:
- 全ペルソナ（4ペルソナ）を並列実行
- Token Budget Strategy に従いリファレンスを選択

### Stage 2: 残りの汎用チェック

ホットスポット以外に対して:
- 単一パスで汎用チェックのみ実施（ペルソナなし）
- Critical レベルの問題のみ検出を目指す
- 対象: 明らかなバグ、セキュリティ脆弱性、リソースリーク

## ホットスポット報告

レビューレポートに以下を含める:
- 特定したホットスポットのリスト（理由付き）
- Stage 1 でレビューした範囲
- Stage 2 でレビューした範囲
- 未レビューのファイル/関数がある場合はその旨を明記
