# Critical Analysis Framework

批判的文書レビューのための分析フレームワーク。すべてのレビュアーペルソナがこのフレームワークを適用する。

## Core Principle: 「本当にそうか？」

すべての主張・結論に対して、以下の姿勢で臨む：

> **「書かれていることを信じない。証拠を求める。」**

## 1. 主張の抽出（Claim Extraction）

### 1.1 明示的主張の特定

文書内で以下のパターンを検出：

| パターン | 例 | 検出方法 |
|---------|-----|---------|
| 結論文 | 「〜である」「〜とする」「〜と判断する」 | 断定的な語尾 |
| 決定事項 | 「〜を採用する」「〜と決定した」 | 決定を示す動詞 |
| 原因特定 | 「〜が原因である」「〜によって発生」 | 因果を示す表現 |
| 評価・判定 | 「問題ない」「十分である」「適切である」 | 評価を示す形容詞 |

### 1.2 暗黙の前提の特定

明示されていないが、文書が成り立つために必要な前提を特定：

- 「〜という前提で」と書かれていない前提
- 業界常識として省略されている事項
- 前工程からの引き継ぎ事項

### 1.3 主張リストの作成

```
主張ID: C-001
主張内容: [具体的な主張]
該当箇所: [文書内の位置]
主張タイプ: [結論/決定/原因特定/評価]
明示/暗黙: [明示的/暗黙的]
```

## 2. 根拠の検証（Evidence Validation）

### 2.1 根拠の5段階評価

各主張の根拠を以下の基準で評価：

| レベル | 評価 | 定義 | 例 |
|--------|------|------|-----|
| **5** | 確実 | 客観的データ・証拠で裏付け | ログデータ、テスト結果 |
| **4** | 信頼性高 | 信頼できる情報源からの引用 | 公式ドキュメント、論文 |
| **3** | 妥当 | 合理的な推論で導出 | 論理的に説明されている |
| **2** | 不十分 | 根拠はあるが不完全 | 一部のデータのみ |
| **1** | 根拠なし | 根拠が示されていない | 「〜と思われる」のみ |

### 2.2 根拠検証チェックリスト

各主張に対して以下を確認：

```
□ 根拠は明示されているか？
□ 根拠の出典は明記されているか？
□ 根拠と主張の間に論理的つながりがあるか？
□ 根拠は検証可能か（再現可能か）？
□ 根拠は最新か（古い情報ではないか）？
□ 根拠は十分な量か（サンプル数、範囲）？
□ 反証となりうるデータは検討されたか？
```

### 2.3 証拠の種類と信頼性

| 証拠タイプ | 信頼性 | 注意点 |
|-----------|--------|--------|
| **直接証拠** | 高 | ログ、スクリーンショット、テスト結果 |
| **間接証拠** | 中 | 状況証拠、推論の連鎖 |
| **証言** | 低〜中 | 「〇〇さんが言った」→確認が必要 |
| **推測** | 低 | 「〜だろう」「〜と思われる」 |
| **仮定** | 要検証 | 前提条件として明示されているか |

## 3. 論理の検証（Logic Validation）

### 3.1 論理的誤謬チェックリスト

| 誤謬 | 説明 | 検出パターン |
|------|------|-------------|
| **因果の誤謬** | 相関を因果と断定 | 「Aの後にBが起きた→AがBの原因」 |
| **循環論法** | 結論を前提に使用 | 「AだからB、BだからA」 |
| **早まった一般化** | 少数事例から一般化 | 「1回成功したから常に成功」 |
| **権威への訴え** | 権威者の発言を根拠 | 「専門家が言ったから」 |
| **感情への訴え** | 感情を根拠 | 「〜べきだ」「当然〜」 |
| **藁人形論法** | 主張を歪めて反論 | 元の主張と違う内容への反論 |
| **滑り坂論法** | 連鎖的悪化を断定 | 「Aすると必ずBになりCになる」 |

### 3.2 論理の連鎖検証

主張がA→B→Cという論理で構成されている場合：

```
検証項目:
1. A→B は妥当か？（各ステップの検証）
2. B→C は妥当か？
3. 途中のステップが省略されていないか？
4. 各ステップに反例はないか？
5. 前提条件は各ステップで維持されているか？
```

### 3.3 因果関係 vs 相関関係

| 関係 | 定義 | 検証方法 |
|------|------|---------|
| **因果関係** | AがBを引き起こす | メカニズムの説明、介入実験 |
| **相関関係** | AとBが同時に発生 | 統計的関連のみ |
| **疑似相関** | 第三の要因Cが両方に影響 | 交絡因子の確認 |

因果関係を主張するには：
- 時間的順序（原因が先）
- メカニズムの説明
- 交絡因子の排除
- 反証可能性

## 4. 危険な表現パターン（Red Flags）

### 4.1 推測・不確実性の表現

| 表現 | リスク | 対応 |
|------|--------|------|
| 「〜と思われる」 | 根拠なしの推測 | 根拠を求める |
| 「おそらく」「たぶん」 | 不確実性 | 確証を求める |
| 「〜かもしれない」 | 可能性の示唆 | 検証されたか確認 |
| 「〜と考えられる」 | 推論 | 推論の根拠を確認 |

### 4.2 過度な確信の表現

| 表現 | リスク | 対応 |
|------|--------|------|
| 「明らかに」 | 根拠なしの断定 | 何が「明らか」か確認 |
| 「当然」「当たり前」 | 暗黙の前提 | 前提の妥当性確認 |
| 「〜に違いない」 | 確証バイアス | 反証の検討確認 |
| 「絶対に」「必ず」 | 過度な確信 | 例外の検討確認 |

### 4.3 曖昧・省略の表現

| 表現 | リスク | 対応 |
|------|--------|------|
| 「など」「等」 | 省略 | 省略されている内容確認 |
| 「一般的に」 | 曖昧な主語 | 具体的な出典確認 |
| 「適切に」「十分に」 | 基準不明 | 基準の明確化要求 |
| 「問題ない」 | 判断基準不明 | 判断根拠確認 |

### 4.4 責任回避の表現

| 表現 | リスク | 対応 |
|------|--------|------|
| 「〜の責任において」 | 責任転嫁 | 実行可能性確認 |
| 「〜次第で」 | 条件付き | 条件の実現性確認 |
| 「想定外の場合は〜」 | 例外処理 | 想定範囲の妥当性確認 |

## 5. トレーサビリティ検証

### 5.1 上流文書との整合性

```
検証項目:
□ 要件定義書の各要件に対応する記述があるか？
□ 要件IDとの紐付けは明確か？
□ 要件の解釈は正しいか？
□ 追加・変更された要件は反映されているか？
□ 削除・除外された要件の理由は明記されているか？
```

### 5.2 下流への影響

```
検証項目:
□ この文書を元に次工程が進められるか？
□ 曖昧さによる解釈の余地はないか？
□ 必要な情報はすべて含まれているか？
□ 判断を要する箇所は明示されているか？
```

## 6. 検証結果の記録

### 6.1 指摘事項フォーマット

```markdown
### [ID] [タイトル]

**重大度**: Critical / Major / Minor / Info

**該当箇所**:
> [文書からの引用]

**問題の種類**:
- [ ] 根拠不足
- [ ] 推測混入
- [ ] 論理飛躍
- [ ] トレーサビリティ欠如
- [ ] 曖昧な表現
- [ ] 確証バイアス
- [ ] その他

**問題の詳細**:
[何が問題か、なぜ問題かを具体的に記述]

**検証質問**:
[この主張を検証するために回答すべき質問]

**推奨アクション**:
[どのように修正・改善すべきか]
```

### 6.2 重大度判定基準

| 重大度 | 基準 | 対応 |
|--------|------|------|
| **Critical** | 根拠がない、または重大な論理誤謬。このまま進めると失敗リスク大 | 必須修正 |
| **Major** | 根拠が不十分、または実害を招く可能性のある曖昧さ | 要修正 |
| **Minor** | 改善が望ましいが、致命的ではない | 推奨修正 |
| **Info** | 参考情報、改善提案 | 任意対応 |

## 7. レビュー実行の心得

### 7.1 基本姿勢

1. **懐疑的に読む**: すべての主張を疑う
2. **証拠を求める**: 「根拠は？」を常に問う
3. **代替案を考える**: 他の説明はないか検討
4. **前提を疑う**: 暗黙の前提を掘り起こす
5. **反証を探す**: 主張を否定する証拠を探す

### 7.2 避けるべき姿勢

- 「専門家が書いたから正しいだろう」
- 「前回も同じだったから問題ない」
- 「細かいことは気にしない」
- 「時間がないから流す」
- 「指摘すると角が立つ」

### 7.3 建設的な指摘の仕方

- 問題の指摘だけでなく、解決の方向性も示す
- 「〜が問題」ではなく「〜を明確にすると良い」
- 質問形式で確認を求める
- 重大度を適切に付与し、優先順位を明確に
