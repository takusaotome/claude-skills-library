# Evidence Evaluation Criteria

文書内の主張・結論を支える根拠（エビデンス）の評価基準。

## 1. Evidence Quality Scale

### 5段階評価基準

| Level | Rating | Definition | Examples | Action |
|-------|--------|------------|----------|--------|
| **5** | Verified | 客観的データ・証拠で検証済み | ログデータ、テスト結果、実測値 | そのまま受け入れ可 |
| **4** | Reliable | 信頼できる情報源からの引用 | 公式ドキュメント、論文、標準規格 | 引用元を確認して受け入れ |
| **3** | Reasonable | 合理的な推論で導出 | 論理的説明あり、複数の状況証拠 | 推論過程を検証 |
| **2** | Insufficient | 根拠はあるが不完全 | 一部のデータのみ、サンプル不足 | 追加根拠を要求 |
| **1** | None | 根拠が示されていない | 「〜と思われる」「おそらく」のみ | 根拠の提示を要求 |

### 評価の具体例

#### Level 5: Verified（検証済み）

```
良い例:
「処理時間は平均 2.3 秒でした（測定データ: 添付資料A、N=1000回、測定日: 2024/12/01）」

評価ポイント:
✓ 具体的な数値
✓ 測定方法の明示
✓ サンプルサイズの明示
✓ データの出典が明確
✓ 再現可能
```

#### Level 4: Reliable（信頼性高）

```
良い例:
「AWS Lambda のコールドスタート時間は 100ms〜1s です（AWS公式ドキュメント: https://...）」

評価ポイント:
✓ 信頼できる情報源
✓ 出典が明記
✓ 検証可能
△ 自環境での実測ではない
```

#### Level 3: Reasonable（妥当）

```
例:
「過去3回の障害はすべてメモリ不足が原因だったため、今回もメモリ不足の可能性が高い」

評価ポイント:
✓ 過去の事例に基づく
✓ 論理的な推論
△ 他の可能性の検討が不十分
△ 今回特有の要因の考慮が必要
```

#### Level 2: Insufficient（不十分）

```
例:
「テスト環境では問題なく動作した」

問題点:
△ テスト環境と本番環境の差異は？
△ テスト範囲は十分か？
△ 負荷条件は？
→ 追加情報が必要
```

#### Level 1: None（根拠なし）

```
悪い例:
「この設計で問題ないと思われる」
「おそらく大丈夫だろう」
「一般的にはこのようにする」

問題点:
✗ 根拠が示されていない
✗ 検証されていない
✗ 再現・確認不可能
→ 根拠の提示を要求
```

## 2. Evidence Type Classification

### Direct Evidence（直接証拠）

| Type | Description | Reliability | Examples |
|------|-------------|-------------|----------|
| **実測データ** | 実際に測定・計測した数値 | 高 | 処理時間、スループット、エラー率 |
| **ログ** | システムが出力した記録 | 高 | エラーログ、アクセスログ、監査ログ |
| **スクリーンショット** | 画面の証拠画像 | 高 | エラー画面、設定画面 |
| **テスト結果** | テスト実行の結果 | 高 | 単体テスト、結合テスト結果 |
| **ソースコード** | 実際のコード | 高 | 問題箇所のコード |

### Indirect Evidence（間接証拠）

| Type | Description | Reliability | Examples |
|------|-------------|-------------|----------|
| **状況証拠** | 直接ではないが関連する証拠 | 中 | タイミングの一致、環境の類似 |
| **類似事例** | 過去の似た事例 | 中 | 以前の障害事例、他システムの事例 |
| **統計データ** | 統計的な傾向 | 中 | 業界平均、ベンチマーク |

### Testimonial Evidence（証言）

| Type | Description | Reliability | Examples |
|------|-------------|-------------|----------|
| **専門家の見解** | 専門家の意見 | 中〜高 | アーキテクトの判断、ベンダー見解 |
| **関係者の証言** | 関係者からの情報 | 低〜中 | 「〇〇さんが言った」 |
| **ユーザー報告** | ユーザーからの報告 | 低〜中 | 障害報告、問い合わせ |

### Inferential Evidence（推論）

| Type | Description | Reliability | Examples |
|------|-------------|-------------|----------|
| **論理的推論** | 論理に基づく推測 | 要検証 | 「AならばB、Bが成り立つ」 |
| **仮説** | 検証前の仮説 | 低 | 「〜ではないか」 |
| **推測** | 根拠の薄い推測 | 低 | 「おそらく〜」 |

## 3. Evidence Verification Checklist

各主張に対して以下を確認：

### 根拠の存在

```
□ 主張に対する根拠が明示されているか？
  - Yes → 次のチェックへ
  - No → Level 1（根拠なし）→ 根拠の提示を要求
```

### 根拠の出典

```
□ 根拠の出典・情報源は明記されているか？
  - Yes → 次のチェックへ
  - No → 出典の明記を要求

□ 情報源は信頼できるか？
  - 公式ドキュメント、論文、標準規格 → 信頼性高
  - 個人ブログ、非公式情報 → 信頼性低、要確認
  - 不明 → 確認を要求
```

### 根拠の十分性

```
□ 根拠は主張を支えるのに十分か？
  - サンプルサイズは適切か？
  - 測定条件は本番に近いか？
  - 異常系・エッジケースも検証されているか？
```

### 根拠の鮮度

```
□ 根拠は最新か？
  - いつ時点のデータ・情報か？
  - 状況が変わっていないか？
  - 古い情報に基づいた判断ではないか？
```

### 根拠の検証可能性

```
□ 根拠は再現・検証可能か？
  - 第三者が同じ方法で確認できるか？
  - 測定方法は明記されているか？
  - データは入手可能か？
```

### 反証の検討

```
□ 反証となりうるデータは検討されたか？
  - 主張に反する事例はないか？
  - 代替説明は検討されたか？
  - 確証バイアスに陥っていないか？
```

## 4. Common Evidence Problems

### 問題パターンと対応

| 問題 | 症状 | 検出方法 | 対応 |
|------|------|---------|------|
| **根拠なし断定** | 「〜である」と断定するが根拠なし | 主張の後に根拠がない | 根拠の提示を要求 |
| **循環論証** | AがBの根拠、BがAの根拠 | 論理の循環をチェック | 独立した根拠を要求 |
| **権威への依存** | 「専門家が言った」のみ | 専門家の根拠を確認 | 具体的根拠を要求 |
| **一般化の誤り** | 少数事例から全体を結論 | サンプルサイズ確認 | 追加事例を要求 |
| **古いデータ** | 状況が変わっている | データの日付確認 | 最新データを要求 |
| **条件の欠落** | 特定条件下のみ有効 | 条件の明示確認 | 条件の明記を要求 |

### 不具合分析レポート特有の問題

| 問題 | 症状 | 対応 |
|------|------|------|
| **再現未確認** | 原因推定のみで再現確認なし | 再現確認を要求 |
| **修正後検証なし** | 修正したが効果検証なし | 効果検証を要求 |
| **影響範囲未確認** | 同様の問題が他にないか未確認 | 横展開確認を要求 |
| **根本原因未特定** | 表面的な原因のみ | 根本原因分析を要求 |
| **因果関係未検証** | 「タイミングが一致」のみ | 因果関係の検証を要求 |

## 5. Evidence Evaluation Examples

### Example 1: 設計文書

**主張**: 「このアーキテクチャで1000リクエスト/秒を処理できる」

| 評価項目 | チェック | 結果 |
|---------|---------|------|
| 根拠の存在 | 性能テスト結果の言及あり | ✓ |
| 出典の明記 | テスト結果ドキュメントへの参照なし | ✗ |
| 十分性 | テスト条件の記載なし | ✗ |
| 検証可能性 | 再現手順の記載なし | ✗ |

**評価**: Level 2（不十分）
**推奨アクション**: テスト結果ドキュメントの参照追加、テスト条件の明記

### Example 2: 不具合分析レポート

**主張**: 「メモリリークが原因でOOMが発生した」

| 評価項目 | チェック | 結果 |
|---------|---------|------|
| 根拠の存在 | ヒープダンプの分析結果あり | ✓ |
| 出典の明記 | ヒープダンプファイルの参照あり | ✓ |
| 再現確認 | 再現テストの実施記録あり | ✓ |
| 因果関係 | メモリ増加とOOM発生の時系列一致 | ✓ |
| 修正検証 | 修正後のメモリ推移データあり | ✓ |

**評価**: Level 5（検証済み）
**コメント**: 適切な根拠と検証がなされている

### Example 3: 要件定義書

**主張**: 「ユーザーは検索機能を最も重視している」

| 評価項目 | チェック | 結果 |
|---------|---------|------|
| 根拠の存在 | 「ヒアリング結果より」との記載 | △ |
| 出典の明記 | ヒアリング議事録への参照なし | ✗ |
| 十分性 | 何人にヒアリングしたか不明 | ✗ |
| 検証可能性 | ヒアリング結果の詳細なし | ✗ |

**評価**: Level 2（不十分）
**推奨アクション**: ヒアリング議事録の参照追加、回答者数・属性の明記

## 6. Evidence Requirement by Document Type

| 文書タイプ | 最低要求レベル | 重点チェック項目 |
|-----------|--------------|-----------------|
| 設計文書 | Level 3以上 | 技術的根拠、非機能要件の検証 |
| 不具合分析 | Level 4以上 | 再現確認、因果関係検証、修正効果確認 |
| 要件定義 | Level 3以上 | ステークホルダー合意、トレーサビリティ |
| 提案書 | Level 3以上 | 実現性根拠、コスト見積もり根拠 |
| テスト結果 | Level 5必須 | 測定データ、再現可能性 |
