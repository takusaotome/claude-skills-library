# Meeting Minutes — Output Format Specification

This document defines the canonical structure, formatting rules, and ambiguity-handling conventions for meeting minutes generated by `meeting-minutes-writer`.

## Canonical Structure

```markdown
### 1. Meeting Information
- **Meeting Name**: [extracted or inferred]
- **Date**: YYYY/MM/DD (Day of Week)
- **Attendees**: comma-separated list (use department names if individuals unknown)

### 2. Action Items

| No. | Action Item | Owner | Priority | Due Date | Notes |
|-----|-------------|-------|----------|----------|-------|
| 1   | ...         | ...   | 🔴/🟡/🟢 | YYYY/MM/DD or TBD | ... |

**Priority Legend**
- 🔴 High — critical / blocking
- 🟡 Medium — important / should do
- 🟢 Low — nice to have / can defer

### 3. Meeting Details

#### Decisions Made
- [Bullet list of concrete decisions made during the meeting]

#### Key Topics and Discussion Points
1. **[Topic 1 title]**
   - Background: [context, why discussed]
   - Key Points: [bullet points of substantive content]

2. **[Topic 2 title]**
   - Background: ...
   - Key Points: ...

#### Notes for Future Meetings / Other
- [Parking-lot items, deferred topics, FYIs]
```

## Required Conventions

### Date Format
- Always `YYYY/MM/DD (Day of Week)` — e.g. `2026/04/30 (Thu)`
- Day of week must be verified with `python3 -c "import datetime; print(datetime.date(YYYY,MM,DD).strftime('%A'))"` — never inferred from memory.
- Time zones: state the zone explicitly if multiple are in play (e.g. `14:00 JST / 01:00 ET`).

### Owner Field
- Use the speaker's actual name as it appears in the source (verbatim).
- If multiple owners, separate with `/` (e.g. `Tanaka / Suzuki`).
- If ownership is unclear, assign to the most relevant speaker and append `(To be confirmed)`.

### Priority Assignment
- 🔴 High: blocks another team / on critical path / explicit "urgent" flag
- 🟡 Medium: tied to a sprint or quarterly goal / explicit "should do"
- 🟢 Low: optimization / nice-to-have / future consideration

### Due Date
- Concrete date if stated → render as `YYYY/MM/DD`
- Relative reference ("next week") → resolve against meeting date and render as concrete date with `* relative` note
- Unstated → `TBD` and add note `Deadline not set`

## Inference Rules

When source material is incomplete:

| Missing Element | Action |
|----------------|--------|
| Meeting name not stated | Infer from main topic — append `* To be confirmed` |
| Date not mentioned | Use `[Date to be confirmed]` |
| Attendees unclear | Use department / role names |
| Owner ambiguous | Assign most-relevant speaker + `(To be confirmed)` |
| Decision unclear | Add to "Notes for Future Meetings" rather than "Decisions Made" |

## Ambiguity Markers

- `* To be confirmed` — inferred or guessed, user should verify
- `[Details unclear]` — content was unintelligible / cut off in source
- `(To be confirmed)` — appended after a name when ownership is uncertain

## Content Prioritization

When summarizing, prioritize in this order:
1. **Decisions** — what was agreed
2. **Action Items** — what will happen next, by whom, when
3. **Discussion Points** — substance of the conversation (collapsed to 2-3 key points per topic)
4. **FYI / Parking Lot** — informational items

## Quality Bar — "3-Minute Readability Test"

A reader unfamiliar with the meeting should be able to extract:
- What was decided
- What they need to do (or what is being done by whom)
- What unresolved questions remain

…in **under 3 minutes** by reading the Action Items + Decisions Made sections only.

## Length Guidance

- Action Items: as many rows as needed (no artificial cap)
- Decisions: usually ≤ 10 bullets
- Discussion Points: ≤ 5 topics, each with ≤ 4 bullets in Key Points
- If a topic warrants more, split into a sub-document and reference it

## Language

- Match the source language by default (Japanese transcript → Japanese minutes; English transcript → English minutes).
- Mixed-language source → use the language with the most coverage; preserve quoted technical terms.

## Example Cell Renderings

```
| 1 | Investigate root cause of API latency spike | Sato | 🔴 | 2026/05/02 (Sat) | Blocking release; report back at next standup |
| 2 | Draft updated SLA proposal | Suzuki / Tanaka | 🟡 | 2026/05/10 (Sun) | Coordinate with Legal |
| 3 | Survey vendors for new monitoring tool | Yamamoto | 🟢 | TBD | Deadline not set |
```
