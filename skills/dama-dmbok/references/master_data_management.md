# Master Data Management（マスターデータ管理）

## 概要

マスターデータ管理（MDM）は、組織全体で共有される重要なビジネスデータを一元管理し、一貫性と信頼性を確保する知識領域です。

---

## マスターデータの定義

> 複数のシステムやビジネスプロセスで共有される、
> ビジネス上重要なエンティティを表すデータ。

### マスターデータの特徴

| 特徴 | 説明 |
|------|------|
| **共有性** | 複数システムで使用される |
| **永続性** | 比較的静的、頻繁に変更されない |
| **重要性** | ビジネス上の核心的な実体を表す |
| **参照性** | トランザクションデータから参照される |

---

## マスターデータの種類

### ドメイン別分類

| ドメイン | マスターデータ例 | 典型的な属性 |
|---------|-----------------|-------------|
| **顧客** | 顧客、連絡先、住所 | 名前、住所、連絡先、セグメント |
| **製品** | 商品、SKU、カテゴリ | 品名、価格、仕様、階層 |
| **組織** | 部門、従業員、拠点 | 名称、階層、役職 |
| **財務** | 勘定科目、コストセンター | コード、名称、階層 |
| **参照** | 国コード、通貨、単位 | コード、名称、有効期間 |
| **サプライヤー** | 仕入先、契約 | 名称、条件、評価 |

## MDMアーキテクチャスタイル

### 4つのスタイル

| スタイル | 説明 | メリット | デメリット |
|---------|------|----------|-----------|
| **Registry** | 各システムのデータを参照 | 低リスク、段階的 | データ統合されない |
| **Consolidation** | 中央に集約、読み取り専用 | 分析に最適 | リアルタイム性低 |
| **Centralized** | 中央で一元管理、配信 | 強い統制 | 導入コスト高 |
| **Coexistence** | 中央と各システムの同期 | バランス型 | 複雑性 |

### スタイル比較図

```
Registry スタイル:
┌────────┐  ┌────────┐  ┌────────┐
│ CRM    │  │ ERP    │  │ Legacy │
│ 顧客   │  │ 顧客   │  │ 顧客   │
└───┬────┘  └───┬────┘  └───┬────┘
    │          │          │
    └──────────┼──────────┘
               ▼
         ┌──────────┐
         │ Registry │  参照のみ
         │ （索引）  │
         └──────────┘

Centralized スタイル:
         ┌──────────┐
         │  MDM Hub │  唯一の真実
         │ (中央)   │
         └────┬─────┘
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
┌────────┐  ┌────────┐  ┌────────┐
│ CRM    │  │ ERP    │  │ Legacy │
└────────┘  └────────┘  └────────┘
```

### スタイル選定ガイド

| 状況 | 推奨スタイル |
|------|-------------|
| 段階的に導入したい | Registry |
| 分析用途が主 | Consolidation |
| 強い統制が必要 | Centralized |
| 既存システムを維持 | Coexistence |

## MDMプロセス

### ゴールデンレコードの作成

```
ソースシステム           マッチング・マージ           ゴールデンレコード

┌──────────────┐                                ┌──────────────┐
│ CRM          │                                │              │
│ 田中太郎     │──┐                            │ 田中太郎     │
│ 東京都新宿区 │  │    ┌──────────────┐        │ 東京都新宿区 │
└──────────────┘  │    │              │        │ 品川本町1-1  │
                  ├───▶│  マッチング  │───────▶│              │
┌──────────────┐  │    │  マージ      │        │ 顧客ID: C001 │
│ ERP          │  │    │              │        │ 信頼度: 95%  │
│ タナカタロウ │──┘    └──────────────┘        │              │
│ 品川本町1-1  │                                └──────────────┘
└──────────────┘
```

### マッチングルール

| ルールタイプ | 説明 | 例 |
|-------------|------|-----|
| **完全一致** | 値が完全に同じ | 顧客コード一致 |
| **ファジー一致** | 類似度で判定 | 会社名 85%以上 |
| **複合ルール** | 複数条件の組み合わせ | 電話番号 AND 郵便番号 |
| **確率的** | 機械学習による判定 | 総合スコア 90%以上 |

### サバイバーシップルール

どのソースの値を採用するかを決定するルール:

| ルール | 説明 | 例 |
|--------|------|-----|
| **信頼度** | ソースの信頼性で選択 | CRM > ERP > レガシー |
| **最新性** | 更新日が新しいものを採用 | 最新のソースを優先 |
| **完全性** | NULL以外を優先 | 値があるものを採用 |
| **頻度** | 最も多いものを採用 | 多数決 |
| **ルールベース** | 属性ごとにソース指定 | 住所はCRM、与信はERP |

## MDM実装アプローチ

### 段階的アプローチ

```
Phase 1: 基盤構築（3-6ヶ月）
├── 対象ドメインの選定（1つから開始）
├── 現状データ分析
├── データモデル設計
├── ツール選定・導入
└── パイロット実施

Phase 2: 展開（6-12ヶ月）
├── マッチング・マージルール確立
├── ガバナンスプロセス構築
├── システム連携開発
├── データクレンジング
└── 対象ドメイン拡大

Phase 3: 成熟（12-24ヶ月）
├── 全ドメイン展開
├── 自動化推進
├── KPI測定・改善
└── 継続的な品質向上
```

### 成功要因

1. **スポンサーシップ**: 経営層の支持
2. **明確なスコープ**: 1ドメインから開始
3. **データガバナンス**: オーナーシップの明確化
4. **段階的アプローチ**: 小さく始めて拡大
5. **変更管理**: 組織的な変革管理

## 品質管理

### MDM固有の品質指標

| 指標 | 説明 | 計算方法 |
|------|------|----------|
| **重複率** | 重複レコードの割合 | 重複数 / 総レコード数 |
| **一致率** | マッチング成功率 | 一致数 / 候補ペア数 |
| **同期遅延** | 配信の遅延時間 | 更新時刻 - 配信時刻 |
| **採用率** | ゴールデンレコード利用率 | 参照数 / 総参照数 |
| **精度** | マッチング精度 | (TP + TN) / 総数 |
