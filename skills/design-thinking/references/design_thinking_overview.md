# Design Thinking Overview

## 歴史と背景

Design Thinking（デザイン思考）は、デザイナーが問題を解決する際の思考プロセスを体系化したものです。

### 起源
- **1960年代**: デザイン手法論の研究が始まる
- **1991年**: IDEO設立、デザイン思考の実践を開始
- **2005年**: Stanford d.school設立、教育への展開
- **2008年**: Tim Brown「Design Thinking」をHBRで発表

### 主要な提唱者
- **David Kelley**: IDEO創設者、Stanford d.school創設者
- **Tim Brown**: IDEO CEO、「Change by Design」著者
- **Roger Martin**: トロント大学Rotman School、「The Design of Business」著者

## Design Thinkingの本質

### 従来のアプローチとの違い

| 従来のアプローチ | Design Thinking |
|-----------------|-----------------|
| 問題を分析して解決 | 問題を再定義して解決 |
| 論理・分析重視 | 共感・創造性重視 |
| 正解を求める | 複数の可能性を探索 |
| 計画→実行 | 試作→学習→改善 |
| 専門家が決める | ユーザーと共創 |

### 3つの思考モード

│  │   発散思考   │ Divergent Thinking                   │

│  └──────────────┘                                       │

│  ┌──────────────┐                                       │
│  │   収束思考   │ Convergent Thinking                  │

## 5つのフェーズ

### フェーズ概要

```
Phase 1: EMPATHIZE（共感）
   │  └─ ユーザーの世界を理解する
   ▼
Phase 2: DEFINE（定義）
   │  └─ 本当の問題を見つける
   ▼
Phase 3: IDEATE（発想）
   │  └─ たくさんのアイデアを出す
   ▼
Phase 4: PROTOTYPE（試作）
   │  └─ アイデアを形にする
   ▼
Phase 5: TEST（検証）
      └─ ユーザーから学ぶ
```

### 反復的プロセス

Design Thinkingは直線的ではなく、反復的です：

- テスト結果によっては、前のフェーズに戻る
- 新しい洞察が得られたら、問題定義を見直す
- プロトタイプを改良しながら、複数回テストする

## Design Thinkingのマインドセット

### 7つの原則

1. **Human-Centered（人間中心）**
   - すべてはユーザーの理解から始まる
   - 技術や事業の都合ではなく、人間のニーズを優先

2. **Empathy（共感）**
   - ユーザーの立場で考える
   - 観察と傾聴を重視

3. **Embrace Ambiguity（曖昧さを受け入れる）**
   - 最初から答えがわからなくても良い
   - 不確実性の中で探索する

4. **Bias Toward Action（行動重視）**
   - 考えるより、まず作る
   - 議論より、実験

5. **Radical Collaboration（徹底的な協働）**
   - 多様な専門性を持つチームで取り組む
   - 異なる視点を歓迎する

6. **Fail Early, Fail Often（早く、たくさん失敗する）**
   - 失敗は学びの機会
   - 小さく素早く失敗する

7. **Show Don't Tell（見せる、語らない）**
   - 言葉より、ビジュアル
   - 説明より、プロトタイプ

## 適用領域

### 適している課題

- **Wicked Problems（複雑な問題）**
  - 明確な正解がない
  - 多くのステークホルダーが関与
  - 状況が常に変化

- **イノベーション創出**
  - 既存の枠組みを超えた発想が必要
  - 顧客が気づいていないニーズを発見

- **顧客体験の改善**
  - サービス・製品のUX向上
  - カスタマージャーニーの最適化

### 適さない課題

- 明確な正解がある問題
- 緊急対応が必要な問題
- 技術的な制約が厳しい問題
- ユーザーリサーチができない状況

## 組織への導入

### 導入レベル

| レベル | 説明 | 活動例 |
|-------|------|--------|
| 個人 | 個人の問題解決力向上 | ワークショップ参加 |
| チーム | プロジェクト単位での適用 | スプリント実施 |
| 部門 | 部門の標準プロセス化 | デザインチーム設置 |
| 組織 | 企業文化としての定着 | 全社教育、評価制度 |

### 成功要因

1. **経営層のコミットメント**
2. **実験を許容する文化**
3. **多様性のあるチーム編成**
4. **十分な時間とリソース**
5. **実際のユーザーへのアクセス**
