# Define Methods（問題定義フェーズの手法）

## 概要

問題定義フェーズでは、共感フェーズで得た洞察を統合し、**解くべき問題**を明確にします。

> 「正しい問題を見つければ、解決策は自ずと見えてくる」

---

## 1. ペルソナ

### 1.1 ペルソナとは

実際のユーザーリサーチに基づいて作成する、**典型的なユーザー像**です。

**目的:**
- チーム全員が共通のユーザー像を持つ
- 「自分だったら」ではなく「この人だったら」で考える
- 設計判断の基準になる

### 1.2 ペルソナの構成要素

```markdown
## 基本プロフィール
- 名前（架空）
- 年齢、性別
- 職業、役職
- 家族構成
- 居住地

## 背景・コンテキスト
- 1日の過ごし方
- 使っているデバイス・サービス
- 情報収集方法
- 決済手段

## ゴール（Goals）
- 機能的ゴール: 何を達成したいか
- 感情的ゴール: どう感じたいか
- 社会的ゴール: どう見られたいか

## フラストレーション（Frustrations）
- 現状の課題
- 諦めていること
- 我慢していること

## 行動特性
- 意思決定スタイル（慎重/即断）
- 新しいものへの態度（早期採用/保守的）
- 価格感度

## 代表的な発言
「〇〇〇〇〇」
```

### 1.3 良いペルソナの条件

```
✓ 具体的で生き生きとしている
✓ 実際のリサーチに基づいている
✓ チームが共感できる
✓ 設計判断に使える粒度
✗ 理想化されすぎていない
✗ 複数の人を1人に混ぜていない
```

## 2. カスタマージャーニーマップ

### 2.1 ジャーニーマップとは

ユーザーが目標を達成するまでの**体験の流れ**を時系列で可視化したもの。

### 2.2 基本構成

```
┌──────────┬─────────┬─────────┬─────────┬─────────┬─────────┐
│  ステージ │  認知   │  検討   │  購入   │  利用   │  推奨   │
├──────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│  行動    │         │         │         │         │         │
├──────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│タッチ    │         │         │         │         │         │
│ポイント  │         │         │         │         │         │
├──────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│  思考    │         │         │         │         │         │
├──────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│  感情    │  😊     │   😐    │   😢    │   😊    │   😊    │
│ カーブ   │    ＼   │  ／ ＼  │  ／     │         │         │
├──────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│ペイン    │         │         │         │         │         │
│ポイント  │         │         │         │         │         │
├──────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│  機会    │         │         │         │         │         │
└──────────┴─────────┴─────────┴─────────┴─────────┴─────────┘
```

### 2.3 作成のステップ

1. **ペルソナを選ぶ**: 誰の旅か
2. **ゴールを決める**: 何を達成するための旅か
3. **ステージを定義**: 大きな段階を区切る
4. **行動を書く**: 各ステージで何をするか
5. **タッチポイント**: どこで接点があるか
6. **思考・感情**: 何を考え、どう感じるか
7. **ペインポイント**: どこで困るか
8. **機会**: どこを改善できるか

### 2.4 活用のポイント

- **感情カーブの谷**に注目 → 改善機会
- **タッチポイントの断絶**に注目 → シームレス化の機会
- **思考と行動の矛盾**に注目 → 隠れたニーズ

## 3. POV（Point of View）文

### 3.1 POV文とは

洞察を1文で表現する**問題定義ステートメント**。

### 3.2 フォーマット

```
[ユーザー] は [ニーズ] を必要としている。
なぜなら [インサイト] だから。
```

### 3.3 良いPOV文の例

```
【悪い例】
「ユーザーは使いやすいアプリを必要としている」
→ 抽象的すぎる、インサイトがない

【良い例】
「忙しい共働きの親は、栄養バランスの良い夕食を30分以内に
用意する方法を必要としている。
なぜなら、子供の健康を大切にしつつも、自分の時間も
確保したいと感じているから。」
→ 具体的、ユーザーの感情・価値観が含まれる
```

### 3.4 POV作成のコツ

| 要素 | ポイント |
|------|----------|
| ユーザー | 具体的な属性を含める |
| ニーズ | 行動ではなく、深層のニーズ |
| インサイト | 意外性のある発見 |

## 4. HMW（How Might We）質問

### 4.1 HMWとは

POVを**アイデア発想可能な問い**に変換する手法。

```
HMW = 「どうすれば〇〇できるだろうか？」
```

### 4.2 変換の手順

```
POV:
「忙しい共働きの親は、栄養バランスの良い夕食を30分以内に
用意する方法を必要としている。」

↓ HMWに変換

HMW 1: どうすれば調理時間を半分にできるか？
HMW 2: どうすれば子供と一緒に料理を楽しめるか？
HMW 3: どうすれば栄養士のアドバイスを手軽に得られるか？
HMW 4: どうすれば買い物の手間をなくせるか？
HMW 5: どうすれば「手抜き」という罪悪感をなくせるか？
```

### 4.3 HMWの粒度

```
【広すぎる】
「どうすれば世界を良くできるか？」
→ 発想の方向が定まらない

【狭すぎる】
「どうすればボタンの色を青にできるか？」
→ 解決策を示唆してしまう

【ちょうど良い】
「どうすれば初めての人でも迷わず購入できるか？」
→ 複数のアイデアが出せる粒度
```

### 4.4 HMW生成のテクニック

| テクニック | 例 |
|-----------|-----|
| ポジティブに | 「問題を解決」→「より良くする」 |
| 逆転する | 「待ち時間を減らす」→「待ち時間を楽しくする」 |
| 仮定を疑う | 「店舗が必要」→「店舗なしでできるか？」 |
| 形容詞を変える | 「速く」→「楽しく」「安心して」 |
| リソースを活用 | 「ユーザー自身が教える」 |

## 5. インサイトステートメント

### 5.1 インサイトとは

観察と分析から得られた**意外性のある発見**。

```
観察: 「ユーザーは説明書を読まない」
分析: 「なぜ？」
インサイト: 「説明書を読む＝自分は賢くない、と感じるから」
```

### 5.2 良いインサイトの特徴

```
✓ 意外性がある（当たり前ではない）
✓ 共感できる（なるほど！と思える）
✓ 行動可能（何かできそう）
✓ 証拠がある（リサーチに基づく）
```

### 5.3 インサイトの見つけ方

1. **矛盾を探す**: 言っていることとやっていることの違い
2. **代替行動を探す**: 公式の方法を使わない理由
3. **感情を探す**: 喜び、苛立ち、諦めの瞬間
4. **習慣を探す**: 無意識の行動パターン
