# Prototype Methods（プロトタイプフェーズの手法）

## 概要

プロトタイプフェーズでは、アイデアを**素早く形にして、学びを得る**ことが目的です。

> 「If a picture is worth a thousand words, a prototype is worth a thousand meetings.」
> — IDEO

---

## 1. プロトタイプの原則

### 1.1 4つの原則

```
┌─────────────────────────────────────────────────────────┐
│              プロトタイプ 4つの原則                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 早く作る（Start Building）                          │
│     考えるより、作る。完璧を目指さない。                 │
│                                                         │
│  2. 捨てる覚悟（Don't Fall in Love）                    │
│     プロトタイプに固執しない。目的は学び。               │
│                                                         │
│  3. 1つの問いに答える（Answer One Question）            │
│     「これで何を学びたいか」を明確に。                   │
│                                                         │
│  4. ユーザーと共に（Build With Users in Mind）          │
│     内部確認用ではなく、テスト用として作る。             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 忠実度（Fidelity）の選択

| 忠実度 | 特徴 | 用途 | 時間 |
|--------|------|------|------|
| **Lo-Fi（低）** | 手書き、紙 | コンセプト検証 | 分〜時間 |
| **Mid-Fi（中）** | デジタル、モノクロ | 構造検証 | 時間〜日 |
| **Hi-Fi（高）** | 実物に近い | 細部検証 | 日〜週 |

**原則**: できるだけLo-Fiから始める

## 2. ペーパープロトタイピング

### 2.1 概要

紙とペンでUIを描き、ユーザーに「使ってもらう」手法。

### 2.2 必要なもの

```
- A4用紙（スマホサイズにカットすると臨場感UP）
- 太いマーカー（細部を描きすぎないため）
- 付箋（差し替え可能なパーツ用）
- ハサミ
- 透明シート（オーバーレイ表現用）
```

### 2.3 作り方

```
1. 主要画面を描く
   - 1画面1枚
   - ボタン、テキスト、画像をシンプルに
   - 色は使わない（または最小限）

2. インタラクティブ要素を付箋で
   - ドロップダウンメニュー
   - モーダルウィンドウ
   - キーボード

3. 画面遷移を準備
   - ユーザーがタップしたら次の画面を出す
   - 「これをタップすると...」と言いながら差し替え

4. テストの進め方
   - 「〇〇したいとしたら、どうしますか？」
   - ユーザーが指でタップしたら、画面を差し替える
   - 反応を観察、質問する
```

### 2.4 コツ

```
✓ 細かく描きすぎない（太いペンを使う理由）
✓ 「未完成」であることを伝える
✓ テスト中に改良してOK
✓ 複数バージョンを用意してもOK
```

## 3. ストーリーボード

### 3.1 概要

体験の流れを**漫画形式**で描き、ユーザージャーニーを可視化。

### 3.2 構成

```
┌─────────────────────────────────────────────────────────┐
│  タイトル: [シナリオ名]                                  │
│  ペルソナ: [ユーザー名]                                  │
├─────────┬─────────┬─────────┬─────────┬─────────┬───────┤
│ Scene 1 │ Scene 2 │ Scene 3 │ Scene 4 │ Scene 5 │ ...   │
│         │         │         │         │         │       │
│[スケッチ]│[スケッチ]│[スケッチ]│[スケッチ]│[スケッチ]│       │
│         │         │         │         │         │       │
├─────────┼─────────┼─────────┼─────────┼─────────┼───────┤
│ 状況説明 │ 状況説明 │ 状況説明 │ 状況説明 │ 状況説明 │       │
├─────────┼─────────┼─────────┼─────────┼─────────┼───────┤
│ セリフ  │ セリフ  │ セリフ  │ セリフ  │ セリフ  │       │
│（思考）  │（思考）  │（思考）  │（思考）  │（思考）  │       │
└─────────┴─────────┴─────────┴─────────┴─────────┴───────┘
```

### 3.3 6〜8コマで描く

1. **起点**: 問題が発生する状況
2. **気づき**: ソリューションとの出会い
3. **行動**: ソリューションを使う
4. **山場**: 核心的な価値を体験
5. **結果**: 問題が解決
6. **エンディング**: 良い状態

### 3.4 用途

- チームでアイデアの理解を揃える
- ステークホルダーへの説明
- 体験全体を俯瞰して検証

## 4. ワイヤーフレーム

### 4.1 概要

画面の**構造とレイアウト**をモノクロで表現。

### 4.2 ツール

| ツール | 特徴 | 難易度 |
|--------|------|--------|
| 紙とペン | 最速 | ★☆☆ |
| Balsamiq | 手書き風、共有しやすい | ★★☆ |
| Figma | 本格的、コラボ可能 | ★★★ |
| Miro/FigJam | ホワイトボード型 | ★★☆ |

### 4.3 ワイヤーフレームの要素

```
┌─────────────────────────────────────┐
│ [Logo]              [Nav] [Nav] [☰] │  ← ヘッダー
├─────────────────────────────────────┤
│                                     │
│  ┌─────────────────────────────┐   │
│  │                             │   │  ← ヒーローエリア
│  │      [Hero Image/Text]      │   │
│  │                             │   │
│  └─────────────────────────────┘   │
│                                     │
│  ┌───────┐ ┌───────┐ ┌───────┐     │
│  │ Card  │ │ Card  │ │ Card  │     │  ← コンテンツ
│  │ [img] │ │ [img] │ │ [img] │     │
│  │ text  │ │ text  │ │ text  │     │
│  └───────┘ └───────┘ └───────┘     │
│                                     │
│        [ CTA Button ]               │  ← CTA
│                                     │
├─────────────────────────────────────┤
│ [Footer links]        [Social]      │  ← フッター
└─────────────────────────────────────┘
```

### 4.4 注意点

- **色を使わない**: グレースケールで
- **リアルなテキスト**: 「Lorem ipsum」は避ける
- **コンポーネントの優先順位**を意識

## 5. クリッカブルプロトタイプ

### 5.1 概要

画面同士をリンクで繋ぎ、**実際に操作できる**プロトタイプ。

### 5.2 ツール

| ツール | 特徴 | 無料枠 |
|--------|------|--------|
| Figma | デファクトスタンダード | あり |
| InVision | シンプル | あり |
| Adobe XD | Adobe連携 | あり |
| Principle | アニメーション強い | なし |
| ProtoPie | 高度なインタラクション | あり |

### 5.3 作り方（Figma例）

```
1. 画面を作成
   - フレームを追加（iPhone 14など）
   - UIコンポーネントを配置

2. プロトタイプモードに切り替え
   - 右上の「Prototype」タブ

3. インタラクションを設定
   - ボタンをクリック → 遷移先画面にドラッグ
   - トリガー: On Click / On Hover / On Drag
   - アクション: Navigate / Open Overlay / Scroll To

4. プレビュー・共有
   - 再生ボタンでプレビュー
   - 「Share」でURLを発行
```

## 6. 物理プロトタイプ

### 6.1 概要

デジタルではなく、**物理的なモノ**を作る。

### 6.2 素材

| 素材 | 用途 | 調達 |
|------|------|------|
| 段ボール | 筐体、形状確認 | 無料 |
| フォームコア | 軽量な構造物 | 100均 |
| 粘土 | 形状の自由度高 | 100均 |
| LEGO | モジュラーな構造 | 購入 |
| 3Dプリント | 精密な形状 | サービス利用 |

### 6.3 「Wizard of Oz」プロトタイプ

```
ユーザーには本物に見えるが、裏では人が操作している。

例:
・音声アシスタント → 裏でオペレーターが回答
・自動推薦システム → 裏で人が選んでいる
・AIチャットボット → 裏で人がタイピング

目的:
・高度な技術なしで体験を検証
・「もし〇〇ができたら」を先にテスト
```

## 7. ロールプレイ

### 7.1 概要

サービス体験を**演じて**検証する手法。

### 7.2 進め方

```
1. シナリオを決める
   - いつ、どこで、誰が、何を

2. 役割を決める
   - ユーザー役
   - サービス提供者役
   - 観察者

3. 小道具を用意
   - ペーパープロトタイプ
   - 名札、衣装

4. 演じる
   - 最初から最後まで通す
   - 途中で止めてディスカッションもOK

5. 振り返り
   - 気づいたこと
   - 違和感のあった点
   - 改善アイデア
```

## 8. プロトタイプ選択ガイド

### 8.1 検証したいこと別

| 検証したいこと | 適したプロトタイプ |
|---------------|-------------------|
| コンセプトの魅力 | ストーリーボード |
| UIの使いやすさ | ペーパープロト → ワイヤーフレーム |
| 体験の流れ | クリッカブルプロト |
| 物理的な操作感 | 物理プロトタイプ |
| サービス全体 | ロールプレイ |

### 8.2 時間別

| 時間 | 選択肢 |
|------|--------|
| 30分 | スケッチ、ストーリーボード（簡易） |
| 2時間 | ペーパープロト、簡易ワイヤーフレーム |
| 1日 | クリッカブルプロト、詳細ワイヤーフレーム |
| 3日 | Hi-Fiプロト、物理プロトタイプ |
