# Test Methods（テスト・検証フェーズの手法）

## 概要

テストフェーズでは、プロトタイプを**ユーザーに試してもらい、学びを得る**ことが目的です。

> 「あなたのソリューションを売り込むな。学びを得ろ。」

---

## 1. テストの心構え

### 1.1 DO と DON'T

```
DO（すべきこと）:
✓ 観察する、聞く
✓ 「なぜそうしましたか？」と質問
✓ 沈黙を恐れない
✓ メモを取る（または録画）
✓ ユーザーの行動と発言の矛盾に注目
✓ 感謝を伝える

DON'T（避けること）:
✗ 解決策を説明しすぎる
✗ ユーザーを誘導する
✗ 防御的になる
✗ 「それは仕様です」と言う
✗ フィードバックを否定する
✗ 言い訳をする
```

### 1.2 バイアスを避ける

| バイアス | 説明 | 対策 |
|---------|------|------|
| **確証バイアス** | 自分の仮説を支持する情報ばかり集める | 反証を積極的に探す |
| **誘導質問** | 答えを含んだ質問をする | オープンエンド質問を使う |
| **社交辞令** | ユーザーが気を使って褒める | 行動を観察する |
| **サンプルバイアス** | 特定の層だけテストする | 多様なユーザーを含める |

## 2. ユーザビリティテスト

### 2.1 概要

ユーザーに**タスクを実行してもらい**、使いやすさを検証。

### 2.2 準備

```markdown
## 1. 目的を明確に
- 何を学びたいか？
- どんな問いに答えたいか？

## 2. 参加者をリクルート
- 3〜5人で十分（85%の問題を発見）
- ターゲットユーザーに近い人

## 3. タスクを設計
- 具体的で達成可能
- 答えを含まない

## 4. 環境を準備
- 静かな場所
- 録画機材（同意を得て）
- 観察者の位置
```

### 2.3 タスク例

```
【悪い例】
「検索機能をテストしてください」
→ 何をすればいいかわからない

【良い例】
「あなたは来週末、友人の誕生日プレゼントを探しています。
予算は3,000円以内で、友人は30代女性で読書が好きです。
プレゼントを見つけて、カートに入れるところまでお願いします。」
→ 具体的なシナリオ、明確なゴール
```

### 2.4 進行フロー

```
1. 導入（5分）
   - 目的説明
   - 「プロトタイプのテストであり、あなたのテストではない」
   - 思考発話法の説明
   - 同意取得

2. ウォームアップ（5分）
   - 簡単な自己紹介
   - 関連する普段の行動について質問

3. タスク実行（15-30分）
   - タスクを読み上げる
   - 観察・メモ
   - 適宜質問「今何を考えていますか？」

4. フォローアップ（10分）
   - 全体的な感想
   - 最も困った点
   - 最も良かった点
   - 改善アイデア

5. クロージング（5分）
   - 追加コメント
   - お礼
```

### 2.5 思考発話法（Think Aloud）

```
ユーザーに「思っていることを声に出して」もらう手法。

説明例:
「このテストの間、頭の中で考えていることを
そのまま声に出していただけますか？
例えば、『ここを押せばいいのかな』とか
『これは何だろう』とか、そのままで大丈夫です。」

ファシリテーターの声かけ:
- 「今、何を探していますか？」
- 「何を期待していましたか？」
- 「どう感じましたか？」
```

## 3. A/Bテスト（コンセプトテスト）

### 3.1 概要

複数のバージョンを比較して、**どちらが良いかを判断**。

### 3.2 進め方

```
1. 比較対象を用意
   - バージョンA と バージョンB
   - 1つの要素だけ変える（理想的には）

2. ランダムに提示
   - 順番バイアスを避けるため、A→B と B→A を混ぜる

3. 評価を収集
   - 好み
   - 理解しやすさ
   - 使いやすさ
   - 各項目を5段階評価など

4. 比較・分析
   - 定量: どちらが選ばれたか
   - 定性: なぜそちらを選んだか
```

### 3.3 注意点

- 同時に変える要素は1つに絞る
- 「どちらが好きか」ではなく「なぜか」を聞く
- サンプル数が少ない場合は傾向として捉える

## 4. フィードバックキャプチャグリッド

### 4.1 概要

テスト中の気づきを4象限で整理するフレームワーク。

### 4.2 構成

```
┌─────────────────────────┬─────────────────────────┐
│         Likes           │         Wishes          │
│        良かった点        │         改善点          │
│          (+)            │          (△)            │
│                         │                         │
│ ・ナビゲーションが      │ ・ボタンがわかりにくい  │
│   わかりやすかった      │ ・文字が小さい          │
│ ・色使いが好み          │ ・読み込みが遅く感じる  │
│                         │                         │
├─────────────────────────┼─────────────────────────┤
│        Questions        │          Ideas          │
│          質問           │        アイデア         │
│          (?)            │          (!)            │
│                         │                         │
│ ・この機能の目的は？    │ ・〇〇を追加しては？    │
│ ・なぜこの順番？        │ ・△△と連携できると良い │
│ ・他のユーザーも同じ？  │ ・□□の表示方法を変える │
│                         │                         │
└─────────────────────────┴─────────────────────────┘
```

### 4.3 使い方

1. テスト中にリアルタイムで記入
2. 付箋を使ってチームで共有
3. パターンを見つける（複数人から同じ指摘があるか）

## 5. NPS（Net Promoter Score）

### 5.1 概要

「他の人に勧めるか」で満足度を測定。

### 5.2 質問

```
「このサービス/製品を友人や同僚に勧める可能性は
どのくらいですか？」

0 ─────────────────────────────────────── 10
全く勧めない                        非常に勧める
```

### 5.3 計算

```
批判者（Detractors）: 0〜6
中立者（Passives）: 7〜8
推奨者（Promoters）: 9〜10

NPS = 推奨者の% - 批判者の%

例:
推奨者 40%、中立者 30%、批判者 30%
NPS = 40 - 30 = +10
```

### 5.4 ベンチマーク

| スコア | 評価 |
|--------|------|
| 70以上 | 素晴らしい |
| 50-69 | 優秀 |
| 30-49 | 良い |
| 0-29 | 改善が必要 |
| 0未満 | 深刻な問題 |

## 6. 学びの統合

### 6.1 テスト後のワークショップ

```
1. 観察結果の共有（15分）
   - 各観察者が気づきを発表
   - 付箋に書いて貼り出す

2. パターンの発見（15分）
   - 似た気づきをグルーピング
   - 繰り返し出てきた問題を特定

3. 重要度の評価（10分）
   - 頻度（何人から指摘があったか）
   - 深刻度（ユーザーへの影響度）

4. 次のアクションを決定（20分）
   - 何を修正するか
   - どのフェーズに戻るか
```

### 6.2 次のステップの判断

| テスト結果 | 次のアクション |
|-----------|---------------|
| 根本的な問題発見 | Phase 2（Define）に戻る |
| アイデアの方向性違い | Phase 3（Ideate）に戻る |
| 細部の改善が必要 | Phase 4（Prototype）を改良 |
| 概ね良好 | 実装フェーズへ進む |

### 6.3 イテレーションの回数

```
一般的なガイドライン:
- 最低3回はテスト→改善のサイクルを回す
- 重大な問題がなくなるまで続ける
- 時間・予算が許す限り繰り返す

注意:
- 1回目で完璧を期待しない
- 各回で「1つ以上」学びがあればOK
```

## 7. リモートテスト

### 7.1 ツール

| ツール | 特徴 | 用途 |
|--------|------|------|
| Zoom/Meet | 画面共有でテスト | モデレート型 |
| Lookback | 録画＋タスク管理 | モデレート型 |
| UserTesting | 大規模、非モデレート | 定量調査 |
| Maze | プロトタイプ分析 | 非モデレート型 |

### 7.2 リモート特有の注意点

```
✓ 事前に環境チェック（カメラ、マイク、ネットワーク）
✓ 画面共有の許可を得る
✓ 顔が見えるようお願いする（表情観察のため）
✓ 沈黙が長くなりやすいので、声かけを増やす
✓ 録画の同意を取る
```

## 8. 成果物のまとめ

| 成果物 | 目的 | 形式 |
|--------|------|------|
| テストレポート | 発見の記録 | ドキュメント |
| フィードバックグリッド | 気づきの整理 | 4象限図 |
| 問題リスト（優先順位付き） | 次のアクション決定 | リスト |
| ハイライトリール | ステークホルダー共有 | 動画クリップ |
| 次のイテレーション計画 | 改善の方向性 | アクションリスト |
