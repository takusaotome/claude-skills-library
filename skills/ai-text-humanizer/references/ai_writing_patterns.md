# AI Writing Patterns Detection Guide

AI（LLM）が生成するテキストに特有の6パターンの検出基準、正規表現、具体例を定義する。

---

## Pattern 1: 視覚的マーカー残存 (Visual Marker Residue)

### Overview

Markdownの装飾記号がプレーンテキストに混入する。人間がプレーンテキストを書く際には通常使わない記号パターン。

**Doc-type aware rule**:

- `email` / `chat`: 見出しマーカー・箇条書き過多を通常どおり加点
- `blog` / `structured`: 構造化Markdownとして扱い、見出しマーカー・箇条書き過多はPattern 1で非加点
- `auto`: スクリプトが文書種別を簡易推定

### Detection Rules

| Marker | Regex (Python) | Weight |
|--------|---------------|--------|
| 太字マーカー | `\*\*[^*]+\*\*` | High |
| 見出しマーカー | `^#{1,6}\s` (行頭) | High (`email/chat`時のみ加点) |
| エムダッシュ | `—` (U+2014) | Medium |
| 全角スラッシュ | `／` | Medium |
| 過剰な括弧 | 文中の `（[^）]+）` が3回以上 | Medium |
| 入れ子引用符 | `「[^」]*『[^』]*』[^」]*」` | Medium |
| 箇条書き記号 | `^[\s]*[-•]\s` (行頭) | Medium (`email/chat`時のみ加点) |

### Examples

**Before (AI):**
> **プロジェクト管理**において、**リスク管理**は非常に重要です。具体的には、**リスクの特定**（潜在的な問題の洗い出し）、**リスクの分析**（影響度の評価）、**リスクの対応**（対策の立案）の3つのステップがあります。

**After (Human):**
> プロジェクト管理でリスク管理は避けて通れない。まずリスクを洗い出して、影響がどのくらいか見積もって、対策を考える。この3つをやるかやらないかで結果が変わる。

### Scoring Logic

- 太字マーカー: 1個につき +3点（最大15点）
- エムダッシュ: 1個につき +2点（最大10点）
- 括弧過多: 1段落3個以上で +5点
- 見出しマーカー: 1個につき +3点（`email/chat`時のみ）
- 箇条書き: 全テキストの30%以上が箇条書きなら +5点（`email/chat`時のみ）
- パターン1の満点: 15点（総合100点中の15%）

## Pattern 2: 単調なリズム (Monotonous Rhythm)

### Overview

文末表現の繰り返し、接続詞の過剰使用、均一な文長によって機械的な印象を与える。

### Detection Rules

| Indicator | Regex / Logic | Weight |
|-----------|--------------|--------|
| 文末「です/ます」連続 | 連続3文以上が同一文末 | High |
| 文末「である」連続 | 連続3文以上が同一文末 | High |
| 接続詞過多 | `^(また|さらに|そして|しかし|一方|加えて|つまり|なお|ただし)[、，]` が全文の20%以上 | High |
| not A but B構文 | `単に.+だけでなく.+も` `〜ではなく〜` | Medium |
| 均一な文長 | 標準偏差が平均文長の20%以下 | Medium |
| 機械的な閉じ方 | `〜と言えるでしょう` `〜ではないでしょうか` `〜が求められます` | Medium |

### Examples

**Before (AI):**
> リモートワークは生産性を向上させます。また、通勤時間の削減にも貢献します。さらに、ワークライフバランスの改善にもつながります。しかし、コミュニケーションの課題もあります。一方で、テクノロジーの進化により、これらの課題は解消されつつあります。

**After (Human):**
> リモートワークで生産性が上がった、というデータはいくつもある。通勤がなくなるのは単純にありがたい。ただ、チームのコミュニケーションは確かに難しくなった。Slackだけでは伝わらないニュアンスがある。ツールは進化してるけど、完全には埋まらない。

### Scoring Logic

- 同一文末3連続: +5点、4連続以上: +8点
- 接続詞開始率20%超: +5点、40%超: +10点
- not A but B構文: 1個につき +3点
- 均一文長: +5点
- パターン2の満点: 20点（総合100点中の20%）

## Pattern 3: マニュアル的構成 (Manual-like Structure)

### Overview

教科書やマニュアルのような体系的すぎる構成。前置きが長く、結論が薄い。

### Detection Rules

| Indicator | Regex / Logic | Weight |
|-----------|--------------|--------|
| 長い前置き | 冒頭100字以上が本題に入らない | High |
| 構成宣言 | `以下では.+について` `本稿では.+を` `ここでは.+を見ていき` | High |
| ステップ表記 | `ステップ[0-9]` `第[一二三四五]に` `[0-9]+つ目は` `まず.+次に.+最後に` | Medium |
| 網羅的列挙 | 3項目以上の並列列挙が2回以上 | Medium |
| 薄い結論 | 末尾が `〜が重要です` `〜が求められています` `〜ではないでしょうか` | High |
| セクション均等 | 各セクションの文字数が±15%以内 | Low |

### Examples

**Before (AI):**
> 近年、デジタルトランスフォーメーション（DX）は多くの企業にとって重要な課題となっています。本稿では、DXの定義、その重要性、成功のための3つのポイント、そして今後の展望について解説します。
>
> まず、DXの定義について確認しましょう。DXとは...

**After (Human):**
> DXという言葉はバズワード化している。でも、実際に成功している企業を見ると、やっていることはシンプルだ。既存業務のデジタル化ではなく、顧客体験そのものを作り変えている。

### Scoring Logic

- 構成宣言: +5点
- 長い前置き: +5点
- ステップ表記: 1個につき +3点（最大9点）
- 薄い結論: +5点
- セクション均等: +3点
- パターン3の満点: 20点（総合100点中の20%）

## Pattern 4: 非コミット姿勢 (Non-committal Stance)

### Overview

断定を避け、あらゆる可能性を網羅しようとする姿勢。責任を負わない書き方。

### Detection Rules

| Indicator | Regex / Logic | Weight |
|-----------|--------------|--------|
| ヘッジ語 | `〜と言えるかもしれません` `〜の可能性があります` `〜とも考えられます` `一概には言えません` | High |
| 強制中立 | `一方で` `他方では` `〜という見方もあります` が1段落に2回以上 | High |
| 弱い否定 | `必ずしも〜ではない` `〜とは限りません` | Medium |
| 断定回避 | `〜傾向にあります` `〜ことが多いです` `一般的に〜` | Medium |
| 免責表現 | `個人の見解であり` `状況によって異なり` `ケースバイケースで` | Medium |
| 両論併記 | `メリットとデメリット` `長所と短所` `利点と欠点` のセット提示 | Low |

### Examples

**Before (AI):**
> リモートワークは生産性の向上に寄与する可能性がありますが、一概には言えません。一方で、コミュニケーションの質が低下する傾向にあるとも考えられます。状況によって異なりますが、適切なバランスを見つけることが重要ではないでしょうか。

**After (Human):**
> リモートワークで生産性は上がる。少なくとも自分のチームではそうだった。コミュニケーションの質は落ちるけど、それは「量」で補える部分が大きい。週2回のオフィス出社がちょうどいい。

### Scoring Logic

- ヘッジ語: 1個につき +3点（最大12点）
- 強制中立: +5点
- 両論併記: +3点
- パターン4の満点: 15点（総合100点中の15%）

## Pattern 5: 抽象語の濫用 (Abstract Word Overuse)

### Overview

具体性のない抽象語を多用し、内容が空疎になる。特に「重要」「適切」「効果的」等の評価語が根拠なく使われる。

### Detection Rules

| Indicator | Regex / Logic | Weight |
|-----------|--------------|--------|
| 空疎な抽象語 | `本質的` `包括的` `体系的` `戦略的` `革新的` `持続可能な` `多角的` `総合的` | High |
| 根拠なき強評価 | `非常に重要` `極めて効果的` `大きな成果` `飛躍的な向上` (根拠なし) | High |
| 実体なき名詞 | `フレームワーク` `アプローチ` `ソリューション` `エコシステム` `パラダイム` (具体説明なし) | Medium |
| 修飾語の空転 | `適切な` `効果的な` `最適な` `理想的な` が具体基準なし | Medium |
| バズワード | `シナジー` `レバレッジ` `スケーラブル` `アジャイル` (文脈不適切) | Low |

### Examples

**Before (AI):**
> 包括的なアプローチにより、持続可能なソリューションを構築することが重要です。戦略的なフレームワークを活用し、効果的な施策を実行することで、飛躍的な成果が期待できます。

**After (Human):**
> やることは3つ。顧客データの一元管理、営業プロセスの標準化、週次レビューの導入。これだけで受注率が15%上がった事例がある。

### Scoring Logic

- 空疎な抽象語: 1個につき +2点（最大10点）
- 根拠なき強評価: 1個につき +3点（最大9点）
- 実体なき名詞: 1個につき +2点
- パターン5の満点: 15点（総合100点中の15%）

## Pattern 6: 定型メタファー (Stock Metaphors)

### Overview

LLMが頻繁に使う定型的な比喩表現。人間の文章では滅多に使われない特有の比喩パターン。

### Detection Rules

| Metaphor | Regex | Weight |
|----------|-------|--------|
| 羅針盤 | `羅針盤` | High |
| 地図 | `(?:道の)?地図(?:を描く\|となる)` | High |
| 設計書・青写真 | `青写真` または `(?:[^\s。、「」『』()（）]+の)?設計書(?:として機能(?:し\|する\|します)?\|となる\|となります\|になる\|になります)` | High |
| 柱 | `(?:の\|大きな\|重要な)柱` | Medium |
| DNA | `(?:企業\|組織\|チーム)の?DNA` | High |
| 車の両輪 | `車の両輪` | High |
| 潤滑油 | `潤滑油` | High |
| エンジン | `(?:成長\|変革\|イノベーション)の?エンジン` | Medium |
| スパイス | `(?:の)?スパイス` | Medium |
| レシピ | `(?:成功\|成長)の?レシピ` | Medium |
| 礎・土台 | `(?:の)?礎\|(?:の)?土台` | Low |
| 架け橋 | `架け橋\|かけ橋\|懸け橋` | Medium |

### Examples

**Before (AI):**
> ビジョンは組織の**羅針盤**であり、企業文化は**DNA**そのものです。この2つは**車の両輪**として機能し、イノベーションの**エンジン**となります。

**After (Human):**
> ビジョンがないと判断に迷う。そして企業文化は一度できると簡単には変わらない。この2つが揃わないとイノベーションは起きない。片方だけではダメだ。

### Scoring Logic

- High重み: 1個につき +4点
- Medium重み: 1個につき +3点
- Low重み: 1個につき +2点
- パターン6の満点: 15点（総合100点中の15%）

Note: 文書タイトルとしての `設計書`（例: `在庫連携バッチ 設計書`）は比喩ではないため、単語単体では検出しない。

## Total Score Calculation

```
Total = Σ(min(pattern_i_score, pattern_i_max)) for i in 1..6
```

各パターンのスコアはそれぞれの満点を上限とする。総合スコアは0-100の範囲。
