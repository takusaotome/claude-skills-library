# Human Writing Techniques

人間っぽく崩す3つの技法。AI生成テキストを自然な文章に変換するための具体的手法。

---

## Technique 1: バランスを崩す（立場を取る）

### Concept

AIは常にバランスの取れた、中立的な文章を書こうとする。人間は立場を持ち、偏りがある。その偏りこそが「人間らしさ」を生む。

### How To

1. **ヘッジ語を削除する**
   - 「〜かもしれません」→ 「〜だ」
   - 「〜の可能性があります」→ 「〜になる」
   - 「〜と考えられます」→ 「〜だと思う」

2. **強い断定を使う**
   - 「重要な要素の一つです」→ 「これが一番大事だ」
   - 「検討に値します」→ 「やるべきだ」
   - 「効果が期待されます」→ 「効果がある」

3. **自分の評価を明示する**
   - 「メリットとデメリットがあります」→ 「メリットの方が大きい」
   - 「様々な意見があります」→ 「自分はこう考える」
   - 「賛否両論あります」→ 「反対意見もわかるが、賛成だ」

4. **優先順位をつける**
   - 「A、B、Cが重要です」→ 「一番大事なのはB。AとCは後でいい」
   - 3つ並列 → 1つに絞って深く

### Example

**Before (Balanced AI):**
> クラウド移行にはコスト削減、スケーラビリティ向上、セキュリティ強化などのメリットがある一方、移行コスト、ベンダーロックイン、データ主権の問題などのデメリットも考慮する必要があります。

**After (Opinionated Human):**
> クラウドに移行すべきだ。コストとスケーラビリティのメリットは明確で、「ベンダーロックイン」を心配して移行しないリスクの方がよほど大きい。データ主権は契約で対処できる。

## Technique 2: 客観を崩す（主観を入れる）

### Concept

AIは客観的・百科事典的な記述を好む。人間の文章には主観的な経験、感情、判断が自然に混じる。

### How To

1. **経験を語る**
   - 「一般的に〜とされています」→ 「実際にやってみると〜だった」
   - 「〜という報告があります」→ 「自分のプロジェクトでは〜だった」
   - 第三者の知見 → 一人称の体験

2. **感覚・印象を入れる**
   - 「効率が向上します」→ 「体感で倍くらい速くなる」
   - 「課題があります」→ 「ここが一番しんどい」
   - 数値的記述 → 感覚的記述を混ぜる

> **Claude使用時の注意**: Claudeがリライトする場合、一人称体験の捏造は「捏造禁止」ルールに抵触する。
> 代わりに、元テキストの主張を断定的な語調に変換する形で主観性を表現する。
> 例: 「一般的に〜とされています」→「〜だ」（体験談の追加ではなく、語調の変換で対応）

3. **判断の根拠を見せる**
   - 「〜が推奨されます」→ 「3社試して〜が一番良かった」
   - 「〜が最適です」→ 「〜を選んだ理由は2つある」
   - 一般論 → 具体的な判断プロセス

4. **疑問や迷いを見せる**
   - 完璧な解説 → 「正直、ここはまだ答えが出ていない」
   - 全知的な語り → 「自分もまだ模索中だが」

### Example

**Before (Objective AI):**
> マイクロサービスアーキテクチャは、サービスの独立デプロイ、技術スタックの多様化、チームの自律性向上などの利点を提供します。ただし、分散システム特有の複雑性、運用コストの増加、デバッグの困難さなどの課題も存在します。

**After (Subjective Human):**
> マイクロサービスに移行して1年経った。独立デプロイできるのは確かに楽だ。ただ、正直に言うと、運用の複雑さは想像以上だった。特に障害時のトレースが地獄で、最初の3ヶ月は毎週のように深夜対応していた。それでも戻りたいとは思わない。

## Technique 3: 論理を崩す（完璧な構造を壊す）

### Concept

AIは常に論理的で体系的な構成を心がける。人間の文章は思考の流れに沿って書かれるため、脱線や飛躍、密度のムラがある。

### How To

1. **自然な脱線を入れる**
   - 本題の途中に関連する余談を挟む
   - 「そういえば〜」「余談だけど〜」
   - 脱線してから本題に戻る

2. **会話的な接続詞を使う**
   - 「しかしながら」→ 「でも」「ただ」
   - 「したがって」→ 「だから」「というわけで」
   - 「一方で」→ 「それはそれとして」
   - 接続詞なしで文をつなぐ

3. **密度に変化をつける**
   - 重要な部分: 深く掘り下げる（200字）
   - 些末な部分: 一言で済ます（20字）
   - 均等な説明 → 傾斜をつける

4. **結論の位置を変える**
   - 冒頭で結論 → 途中で脱線 → 別の角度から補強
   - 時に結論なしで終わる（余韻を残す）
   - 結論を疑問形で投げかける

5. **文体を混ぜる**
   - ですます調と断定調を混ぜる（意図的に）
   - 長い分析文の後に短い感想
   - フォーマルとカジュアルの切り替え

### Example

**Before (Perfectly Structured AI):**
> プロジェクト管理には3つの重要な要素があります。
>
> 第一に、スコープ管理です。プロジェクトの範囲を明確に定義し、スコープクリープを防ぐことが重要です。
>
> 第二に、リスク管理です。潜在的なリスクを早期に特定し、対策を講じることが求められます。
>
> 第三に、ステークホルダー管理です。関係者との適切なコミュニケーションが不可欠です。

**After (Naturally Messy Human):**
> プロジェクト管理で一番大事なのはスコープを守ることだ。これが崩れると全部崩れる。
>
> 実は昨年、あるプロジェクトでクライアントの「ちょっとした追加要望」を安易に受けてしまい、2ヶ月遅延した。たった1つの機能追加が、テスト工数を倍にした。
>
> リスク管理もステークホルダー管理も大事だけど、スコープさえ守れていれば大体なんとかなる。逆に言えば、スコープが崩れたプロジェクトは何をやっても救えない。

## Application Guidelines

### 3技法の適用バランス

- テキストの用途に応じて技法の強度を調整する
- ビジネス文書: Technique 1（立場）を中心に、Technique 3（論理崩し）は控えめに
- ブログ記事: 3技法をバランスよく適用
- SNS投稿: Technique 2（主観）と Technique 3（論理崩し）を強めに
- 学術的文章: Technique 1のみ軽く適用、Technique 2-3は最小限

### 注意事項

- 3技法すべてを同時に最大強度で適用しない（やりすぎると別の不自然さが出る）
- 元テキストの目的と読者を考慮する
- フォーマルな文書では崩しの度合いを抑える
- 「崩す」とは「壊す」ではない。読みやすさと情報の正確さは維持する
